# Custom Mobile GPU Architecture — Ultra-Tiny Phone SoC GPU
*Fully specified, physically realizable, vertically integrated with the MUX switch architecture*

## 2. Design Philosophy

The Pixelion-128 is built around **three hard constraints** that define every design decision:

1. **Zero CPU traffic on hot path** — after initial setup, the CPU can be clock-gated on the GPU render path. The display engine and fabric operate on their own PLLs.
2. **Tile-first, not triangle-first** — on-chip tile store means DRAM bandwidth drops ~8× versus immediate-mode rendering. This is the single largest power saver for a mobile GPU.
3. **Deterministic latency** — every pipeline stage has a fixed, bounded pipeline depth. No dynamic division. Variable-latency operations (texture cache miss) are masked by tile-level thread scheduling, not by global stalls.

## 4. Pipeline Architecture

### 4.1 Command Pipeline (Host → GPU)

```
CPU (App / Vulkan / GLES driver)
    │
    ▼  writes command stream via DMA (1 KB–256 KB rings)
┌─────────────────────────────────────────────────────────┐
│  COMMAND PROCESSOR (CP)                                 │
│                                                         │
│  Ring Buffer A (primary):  248 KB @ 0x9000_0000        │
│  Ring Buffer B (indirect):  8 KB @ 0x9003_E000         │
│  Preempt Buffer:            4 KB @ 0x9003_F000         │
│                                                         │
│  CP fetch: 4 × 64-bit words/cycle (256 B/s)            │
│  Decode: packet parser (Vulkan indirect draw encoded)   │
│  Validation: address range check, privilege check       │
│  Issue: write to peripheral bus → CU / Display / DMA    │
└────────────────┬────────────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────────────┐
│  MICROSEQUENCER (µS)                                    │
│  32-entry scoreboard, 8× parallel issue width           │
│  Handles: context switch, preemption, recovery          │
└─────────────────────────────────────────────────────────┘
```

**CP packet format (64-bit aligned, little-endian):**

```
[63:56]  Packet Type
  0x00: NOP
  0x01: DRAW (indexed / non-indexed)
  0x02: DISPATCH (compute shader)
  0x03: CLEAR (render target)
  0x04: RESOLVE (tile → framebuffer blit)
  0x05: MEM_BARRIER / CACHE_FLUSH
  0x06: EVENT / TIMESTAMP
  0x07: PREEMPT / CTX_SWITCH

[55:48]  Sub-type / Flags
[47:32]  Length in dwords (excluding this header)
[31:0]   Payload pointer (32-bit GPU physical address)
```

### 4.2 Compute Pipeline (Shader Execution)

```
Vertex Shader (optional — geometry shader can bypass)
    │
    ▼
Tessellation (optional, HW tessellator: 2× patch, max 64 vertices/patch)
    │
    ▼  primitive assembler
Geometry Shader (optional, capped at 8 output vertices per primitive)
    │
    ▼  → frustum / occluder culling → **TILE GENERATOR**
```

**Tile Generator** (`tile_gen` block, 0.03 mm²):
- 16×16 pixel tile size (configurable: 8×8 for shadow maps, 32×32 for full-screen)
- Binning result: per-tile list of triangle indices
- Output: tile descriptor written to fabric SRAM (512 bytes per tile @ 1080p, ~240 tiles total)
- Tile descriptor format:

```
Offset  Field                  Bits
─────────────────────────────────────────────────────────
0x00    Tile Origin (x, y)     2 × 11 (0–2047 for 4K panel)
0x04    Primitive Count        16
0x06    Sample Mask (MSAA)     4
0x08    Z-Bounds (near, far)   2 × FP16
0x0C    Color Bounds (RGBA8)  4 × 8 (tight bounds for early-z rejection)
0x10    Varying Stream Offset  24 (L2 address of packed varyings)
0x14    Flags                  [31]=MSAA, [30]=sample_count_log2, [29]=alpha_test
```

**Per-tile execution model:**

```
for each tile in screen:
    1. Load tile descriptor from L2 (HW prefetch, 1 cycle)
    2. Load primitive list (L2 burst, 16–32 B average)
    3. SETUP UNIT: attribute interpolation setup (16-varying max)
       - Edge function coefficients computed in parallel (2 cycles)
    4. SHADING PHASE:
       Schedule threads across 2 CUs (128 total lanes)
       Each warp = 32 threads (1 wavefront = half a CU)
       Max 4 concurrent warps per CU = 8 warps total in flight
       
       Thread state: 32 KB register file per CU
       Per-thread: 256 registers × 4 bytes = 1 KB thread state
       128 threads × 1 KB = 128 KB (fits in RF + L2 spill)
       
       SIMD-8 executes 8 threads in lockstep per CU sub-unit
       Scalar core handles branches, predicate, constants
    5. EARLY-Z / EARLY-STENCIL:
       After vertex shading, before fragment shading
       Hierarchical Z (4-level MIP pyramid, 1 KB on-chip RAM)
       Rejection rate: ~45% average on typical game content
    6. FRAGMENT SHADING:
       Texture L1$ checked first (per-CU, 32 KB, hit rate ~92%)
       L2$ miss → DRAM fetch (128 B cache line, 32 GB/s L2 bus)
    7. BLENDING / OUTPUT:
       Tile-local framebuffer in L2$ (128 KB reserved = enough for
         4 tiles @ 1080p 32-bpp, ping-pong between banks)
       Blending is ROP-level, happens before tile resolve
    8. TILE RESOLVE:
       Tile compressed (RLE for solid-color spans, ~3:1 typical)
       DMA write to framebuffer pool in DRAM (via direct-GPU port,
         bypasses APB, lands in DRAM independently)
```

### 4.3 Fixed-Function Unit Specs

| Unit | Count | Key Specs |
|---|---|---|
| Vertex Fetch Unit | 2 | 2 vertices/cycle, VBO cache 8 KB each |
| Tessellator | 1 | HW fixed-function, 16×16 patch limit, no tessellation factor clamping |
| Geometry Shader | 2 | 8 verts out, 2 prims in, hardware ring buffer |
| Clipper / Culler | 1 per CU pair | 64 prims/clip, trivial accept rate ~70% |
| Setup (triangle walk) | 1 per CU pair | 4 triangles/cycle, edge function engine |
| ROP | 16 (8 per CU tile engine) | 32-bit RGBA8 + 24-bit depth + 8-bit stencil per pixel<br>Blend modes: all OpenGL ES 3.2 + Vulkan blend ops<br>MSAA: 2× and 4× only (no 8× — memory prohibitive on mobile) |
| Texture Unit | 4 (2 per CU) | 4 texel/cycle, max 8K×8K, anisotropic 16:1 max<br>Formats: RGBA8, RGBA16F, RGBA32F, bc1-5, bc7, astc 4×4/6×6/8×8/12×12<br>Sampler states: 16 per CU context (mirrored to L2$ on context switch) |

## 5. Memory Hierarchy

### 5.1 Physical Address Map (GPU-visible)

```
GPU Physical Address Map (32-bit, 4 GB space)
═══════════════════════════════════════════════════════════

0x0000_0000 – 0x07FF_FFFF   DRAM Direct (512 MB)
```

### 5.3 Tile Store Compression

The on-chip tile store (128 KB, shared between the 2 CUs via L2$) uses run-length encoding per scanline:

```
Per scanline header (16 bytes):
  [0]   compressed flag (0=raw, 1=RLE)
  [1]   format (0=RGB888, 1=RGB565, 2=RGBA8888, 3=RGBA16F, 4=DS 24_8)
  [2:3] width_in_pixels (≤ 16 for 16×16 tile)
  [4]   span_count

Per span (variable, 4–16 bytes for typical data):
  [0]   x_start, x_end (4 bits each → max 16 pixels wide)
  [1]   sample_count (1 for no-MSAA, 4 for 4×MSAA = sub-pixel data)
  [2..] compressed pixel data

Average compression: 3:1 on UI, 2.5:1 on games, 1.2:1 on noisy content
Worst case: 1:1 (high-frequency noise, depth buffer)
```

## 6. Shader ISA (Instruction Set Architecture)

A custom scalar + SIMD-8 ISA, designed for low decode complexity and high IPC at low power.

### 6.1 Register File Layout

```
Per-thread register allocation (128 registers × 4 bytes = 512 bytes / thread)

  R0  – R7    : 8 scalar registers (32-bit, read/write)
               used for: loop counters, predicate masks, uniform pointers
  R8  – R23   : 16 attribute varyings (interpolated, read-only post-setup)
  R24 – R31   : 8 private temporaries (32-bit, per-thread scratch)
  R32– R63   : 32 × 32-bit vector data (SIMD-8 packed, 4 bytes per lane)
               element 0 = lane 0 (thread 0, 8, 16, 24, 32, 40, 48, 56)
  R64– R95   : 32 × 32-bit vector data (SIMD-8 packed)
               element 0 = lane 1 (thread 1, 9, 17, 25, 33, 41, 49, 57)
               ... up to R96–R127 for lanes 2–7 (7 more blocks of 32)

Total: 128 × 4 = 512 bytes per thread
128 threads × 512 B = 64 KB per warp slot (CUs have 4 warp slots = 256 KB RF)
Actual implementation: 32 KB per CU register file (4 warps × 128 threads × 512 B would be 256 KB).
Compromise: 4 warps × (4 × 8 × 32-bit) = 16 KB scalar + 16 KB vector shared context,
             thread-local data spills to L1$ on context switch (1 μs latency).
```

> **Why 4 warps, not 8?** 4 warps per CU gives 64 total warps in flight across 2 CUs. With 8× SIMD lanes per CU, peak throughput is achieved when 4 warps hide L2 latency (worst case 40 cycles, fully pipelined at 1 cycle/issue = 40-deep instruction buffer needed). 4 warps × 32 threads × 4-issue = 512 outstanding ops per CU, easily hiding 40-cycle L2 miss.

### 6.2 Instruction Encoding

All instructions are 64-bit (2 × 32-bit halfwords). Packed into 128-bit line (4 instructions per L1$ fetch).

```
[63:56]  Primary Opcode (8 bits = 256 opcodes)
[55:48]  Destination register specifier
[47:40]  Source register specifier 1 (src0)
[39:32]  Source register specifier 2 (src1)
[31:24]  Source register specifier 3 (src2) / Immediate select
[23:16]  Modifier / Swizzle / Predicate
[15:0]   Immediate value / Branch offset (16-bit signed, word-aligned = ×4)

Special cases:
  [15:0] used as literal constant when Modifier[7] = 1 (immediate mode)
  [55:48] repurposed as SIMD lane mask when Predicate[7] = 1
```

### 6.3 ISA Coverage (required for Vulkan 1.3 + GLES 3.3)

| Class | Instructions | Notes |
|---|---|---|
| Scalar ALU | ADD, SUB, MUL (U/S/F), DIV, MOD, MIN, MAX, ABS, NEG, CMP, SEL | DIV is integer 32-cycle, FP 8-cycle; use reciprocal-estimates for real-time |
| Vector ALU | VADD, VSUB, VMUL, VFMADD (FMA3 = a×b+c), VMAX, VMIN, VCMP | FMA: IEEE 754-2008 compliant, single-cycle throughput, 4-cycle latency |
| Special Functions | VSQRT, VRSQRT, VRECIP, VSIN, VCOS | RSQRT/RECIP use 2.8-bit initial approx + Newton-Raphson × 2 (total 4 cycles)<br>Trig: 12-cycle polynomial (minimax, max error < 1 ULP for FP16) |
| Data movement | VMOV, VLD (gather), VST (scatter), VGATHER, VSCATTER | Scatter-gather address must be L1$-resident (256-byte aligned) |
| Memory | LD, ST, ATOMIC_ADD, ATOMIC_CAS, ATOMIC_MIN/MAX | Atomics limited to L2$/DRAM; L1$ cache line must be in Modified state |
| Control | BRANCH, CALL, RET, LOOP, IF/ELSE (predicated blocks) | Branches are warp-synchronous: mask register splits active threads |
| Texture | TEX_SAMPLE, TEX_BIAS, TEX_GRAD, TEX_LOD | 4-texel/cycle throughput, up to 16 anisotropy (clamped per-SamplerState) |
| Tile / Raster | TILE_BEGIN, TILE_EMIT, TILE_DONE, EARLY_Z_TEST | Driver-controlled VS→PS path with explicit early-z disable for alpha-tested |

**SIMD-8 execution note:** Every SIMD instruction has an implied lane-grouping of 8 threads. A `VADD R32, R33, R34` computes `R32[i] = R33[i] + R34[i]` for threads i ∈ {0, 1, …, 7} simultaneously. The other 8 active threads in the same warp are serviced by the second SIMD sub-unit (mirror operations).

## 7. Pipeline Timing (Full GPU Cycle Budget)

### 7.1 Vertex Shading (Tile Binning Stage)

```
Stage                        Cycles  stalls  Comment
──────────────────────────────────────────────────────────────────
Vertex Fetch (2 verts/clk)   ceil(nverts/2)  0     VBO cache hit assumed
Vertex Shader (SIMD-8)       3 waves × 8 cyc 2    32-thread warp, 3 waves=96 verts
Assembly + Cull              4 cycles          1    frustum + backface + small-tri
Tile Bin Assign              2 cycles/tile     0    insert into bin list in L2$
Total per primitive           ~14 cycles avg    (pipelined, 4 prims in flight)
```

### 7.2 Fragment Shading (Pixel Pipe)

```
Stage                        Cycles  stalls  Comment
──────────────────────────────────────────────────────────────────
Setup (edge functions + interp coeff)  3   0   per tile, 1-time cost
Tile Scan: 16×16 = 256 pixels
  ─ Per warp (32 pixels, 2×2 quad coverage)
    Early-Z Test                1           hierarchical Z pyramid lookup
    Quad Reject (z-stencil)     0–1         average 45% rejection rate
    Fragment Shader Invocation  ceil(24/8)=3 × 8 = 24 cycles  (SIMD-8, 8 cyc latency, 1 issue/cycle)
    Texture L1$ fetch            3 (hit) / 12–40 (miss)
    Texture Filter bilinear      2 (pipelined alongside SFU)
    Blending (ROP)               1 cycle/pixel, 16 ROPs = burst 16 pixels/cycle
Tile Resolve (compress RLE)    4 cycles    per tile, DMA burst to DRAM
──────────────────────────────────────────────────────────────────
Total: ~75 cycles/tile tile-to-DRAM  (vs. ~400 cycles for immediate mode without tile cache)
```

### 7.3 Critical Latency Path (worst case)

```
Frame start → tile descriptor fetch → vertex shade → bin → fragment → tile resolve → DRAM complete

Fixed overhead (per frame):
  Command fetch + decode:               20–50 μs  (depends on draw call count)
  CP → pipeline fill:                   8 cycles × 4 issues/cyc = 32 cycles
  Clear + load ops:                     3 μs
────────────────────────────────────────────────
Variable per-tile (256-pixel tile, SIMD-8 fully utilized):
  Vertex + bin:                         14 cycles/tile
  Fragment shade (alpha=reject 0%):     75 cycles/tile
  Resolve + compress:                   4 cycles/tile
────────────────────────────────────────────────
Worst-case frame (1080p = 120 tiles, 100% shading occupancy):
  CPU→Display latency (direct GPU path): 0 μs  (GPU drives display, no CPU compositor)
  Pipeline depth from flip to first pixel on panel: 8.33 ms  (120 Hz frame = 8.33 ms budget)
  All rendering fits in budget with 3.5 ms margin for composition and panel blanking
```

## 8. Display Engine (Direct-GPU Output Path)

The display output engine is wired **directly to the MUX switch fabric XBAR output port A** via a 32-bit or 64-bit internal tile DMA channel. No APB is involved once the session is established.

```
└────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────┐
```

### 8.2 Display Engine MUX Integration

The display engine's output simultaneously feeds **both** the fabric XBAR and an internal scanout FIFO:

```
Display Engine Output
═════════════════════
  Internal Panel path  ─────────────────────────► OUTPUT A (Panel)
  via fabric XBAR      (strict display-only path, GPU locks this port)
                        
  External Display path ────────────────────────► OUTPUT B (USB-C)
  via fabric XBAR       (independent, compositor can route here too)
  
  Under MUX_OWNER control:
    MUX_OWNER=0b01 (GPU owns A): Panel gets GPU-only, USB-C gets ext compositor or HDMI
    MUX_OWNER=0b10 (EXT owns A):  Panel gets nothing (off/desktop mode), USB-C gets GPU
    MUX_OWNER=0b00 (composite):  Panel gets composite, USB-C gets composite
```

The display engine's scanout FIFO generates a **frame-done interrupt** (to fabric → kernel) on every vsync. This is the GPU's primary way to know the panel consumed the tile, and the rate-limiter for the GPU's render-ahead buffer.

## 9. Command Processor & Microsequencer State Machine

The CP is a 4-stage in-order machine:

```
┌──────────────────────────────────────────────────────────────────┐
│  1. FETCH  (reads ring buffer, increments WP via GPU MMIO mirror)│
│     Fetch width: 4 × 64-bit = 256 B/cycle                      │
│     Prefetch: 2 × 256 B lookahead = 512 B in flight            │
├──────────────────────────────────────────────────────────────────┤
│  2. DECODE (packet type dispatch, privilege check)              │
│     Privilege: CP runs in Secure/Priv mode via fabric EL3 gate  │
│     Userspace ioctl packets go through kernel → SMC → CP        │
│     Direct DRAM reservation happens here (command-ring region   │
│     is validated to be within GPU_PRIVATE range 0x1000_0000)    │
├──────────────────────────────────────────────────────────────────┤
│  3. VALIDATE (resource bounds checking)                         │
│     · Vertex buffer: check VA ∈ [0, DRAM_DIRECT_SIZE)           │
│     · Texture: check sampler state ptr ∈ [0x2000_0000, 0x2FFF_FFFF)│
│     · Framebuffer: check attachment ptr in framebuffer pool range│
│     · Tile count: if > 1024, reject with ERROR_TILE_OVERFLOW    │
├──────────────────────────────────────────────────────────────────┤
│  4. ISSUE  (write to peripheral bus)                            │
│     Destination:                                                        │
│       [31:28] = 0x0  →  CU0/CU1 (compute dispatch)                   │
│       [31:28] = 0x1  →  Display Engine (scanout config)              │
│       [31:28] = 0x2  →  DMA engine (tile resolve, VRAM push)         │
│       [31:28] = 0x3  →  µS (microsequencer interrupt)               │
│       [31:28] = 0xF  →  Fabric (direct XBAR config, privilege only) │
└──────────────────────────────────────────────────────────────────┘
```

### 9.1 Context Switching (Preemption)

```
Preempt Trigger Conditions
──────────────────────────
  · Higher-priority process preempts (2-level: game > compositor > background)
  · Thermal emergency (policy engine signals via fabric register interrupt)
  · User-initiated (alt-tab, SWITCH_DISPLAY ioctl)

Preempt Sequence (hardware-automated, < 100 μs):
  1. µS asserts CP_PAUSE, current draw/dispatch completes at next tile boundary
  2. Save CU state:
     · 32 KB register file × 2 CUs → DRAM context save area (256 KB region per context)
     · L1$ tags + dirty bits → L2$ (flush modified lines, no writeback to DRAM yet)
     · Tile bin lists → DRAM
  3. µS loads new context
     · DRAM → register file: 4 μs (256 KB burst, 64 GB/s copy engine)
     · L2$ tags → L1$ tag restore: 1 μs
  4. CP resumes fetching from new ring buffer base
  5. µS deasserts CP_PAUSE

  Total hardware preempt time: 5–10 μs typical, < 100 μs worst case
  (only means the CPU can't switch faster than this without CPU-side buffering)
```

### 9.2 Microsequencer Scoreboard

```
Scoreboard Entry (32 entries total, 8-issue width)
══════════════════════════════════════════════════
Entry field        Bits  Purpose
─────────────────────────────────────────────────
Valid              1     Instruction issued, waiting for read
Src0 ready         1     Register src0 not in flight
Src1 ready         1     Register src1 not in flight
Src2 ready         1     Register src2 (or immediate) ready
Dest written       1     Set at write-back, clears src-dependents
CU target          1     0=CU0 / 1=CU1 (pairs with 2-wide dispatch)
Reorder tag        10    Unique ID for out-of-order complete tracking
─────────────────────────────────────────────────
Total:             14 bits × 32 = 448 bits SRAM

Scheduling policy: WAR-aware, in-order issue, out-of-order complete
Wakeup: source tag CAM lookup (12-entry associative array per CU)
Stall: if < 4 free scoreboard entries, back-pressure to CP fetch
```

## 11. Integration with MUX Switch Fabric

This GPU is designed from the first RTL pass to expose the following **direct-to-fabric** interfaces:

### 11.1 Direct-GPU Port (XBAR Input Port 0)

```
Signal Group: gpu_to_xbar (32-bit per lane, 32 lanes total = 1024-bit bus)
══════════════════════════════════════════════════════════════════════
Lane  [31:0]  — Tile scanout data (RGB888 / RGBA16161616, packed)
Lane  [32]    — H-SYNC (frame start indicator)
Lane  [33]    — V-SYNC (field/frame valid)
Lane  [34]    — DATA_VALID (pixel valid in current cycle)
Lane  [35]    — TILE_START (tile boundary, tile descriptor follows)
Lane  [36]    — TILE_END (end of tile, fabric issues MUX_CREDIT_ACK)
Lane  [37]    — COMPRESSION_TYPE (2-bit: 0=raw, 1=RLE, 2=delta, 3=sparse)
Lane  [38]    — ERROR (frame CRC mismatch, fabric reports to kernel)
Lane  [39:63] — Reserved (lane count expansion to 64 → 2048-bit future)
```

**Timing contract between GPU and Fabric:**
- `clk_fabric` = 4× `clk_gpu` (derived PLL, phase-aligned on GPU boot)
- GPU TX FIFO: 128 deep, runs on `clk_gpu`, crosses to `clk_fabric` synchronously at fabric boundary
- Fabric credits 32 tokens; GPU must acquire credit before asserting `TILE_START`
- Credit acquire time: 0 cycles if available, ≤ 4 cycles if pool exhausted (fabric returns NACK; GPU stalls tile generator)
- If credit pool persistently full (> 1 ms): GPU asserts `GPU_STALL` signal to APB interrupt, kernel `muxd` handles stall recovery

### 11.2 Fabric Register Sideband (AXI-lite slave to GPU)

```
GPU Register Space (mapped at fabric 0x9200_0100 – 0x9200_01FF)
════════════════════════════════════════════════════════════════
This is a 256-byte AXI-lite slave port inside the GPU, exposed to the
fabric's APB bridge. The kernel driver writes these to configure the
GPU→fabric tile stream.

0x00   GPU_TILE_DESC_BASE    RW  [31:0]  Base address of tile descriptor pool in DRAM
0x04   GPU_TILE_DESC_STRIDE  RW  [15:0]  Bytes per tile (typically 64 or 128)
0x08   GPU_FB_POOL_BASE      RW  [31:0]  Framebuffer pool base DRAM address
0x0C   GPU_FB_POOL_STRIDE    RW  [15:0]  Bytes per scanline (stride in bytes)
0x10   GPU_FB_POOL_HEIGHT    RW  [15:0]  Total height in pixels
0x14   GPU_CREDIT_REFILL     RW  [7:0]   Fabric credit tokens to request per vsync
0x18   GPU_STATUS            RO  [31:0]  [0]=GPU_IDLE [1]=TILE_DONE [2]=FB_FAULT
         [3]=CREDIT_EMPTY [4]=XFER_COMPLETE [31:5]=reserved
0x1C   GPU_IRQ_EN            RW  [7:0]   Enable GPU→fabric→kernel IRQs
0x20   GPU_IRQ_STATUS        W1C [7:0]  Write-1-clear interrupt flags
0x24   GPU_TJ_SENSOR         RO  [15:0]  GPU junction temperature, 0x8000 = 128 °C
         (raw ADC, driver scales: temp(°C) = raw / 256, calibrated at test)
0x28   GPU_PWR_STATE         RW  [2:0]   0=on 1=retention 2=off (power-gate request)
```

## 13. Real-World Performance Model (1080p Gaming)

Assumptions:
- 1920 × 1080 @ 16×9, 32-bpp RGBA8888 framebuffer
- 120 Hz refresh (8.33 ms frame budget)
- Average game: 40K triangles/frame, ~200 draw calls
- Post-processing: 4× MSAA, 2 render targets (color + depth), 1 post-process pass

### 13.1 Throughput Budget (constrained by worst stage)

```
Bottleneck Analysis
═══════════════════

Stage                                    Throughput       Occupancy
────────────────────────────────────────────────────────────────────
Command fetch + decode (200 draw calls)  ~8 μs            < 0.1 ms budget
Vertex shading (40K tris → 120K verts):
  120K verts / (8 warp/cyc × 8 cyc) = 1875 cycles/core
  @ 1.5 GHz = 1.25 μs × 2 CUs = 1.25 μs            1.3 ms budget
Tile binning (120 tiles):                120 × 14 cyc      0.4 ms budget
Fragment shading (120 tiles × 256 px):
  Accept rate (120 Hz): ~85% pass early-Z:
  Effective 256 × 0.85 × 120 tiles = 26.1K fragments
  26.1K / (8 warp/cyc × 8 cyc) = 408 cyc/warp × 2 = 816 cycles per
    256-px tile (worst case, fully shaded, no overdraw): 16 × 8 cyc = 128
    So: 816 × 4 tiles/CU = 3264 cycles per tile-slot
  × 120 tiles / 2 CUs = 6.0 μs             7.0 ms budget
ROP + tile resolve (120 tiles):
  120 × 4 cyc + 120 × 4 cyc compress = 960 cyc → 0.64 μs     < 1 ms (hidden)
DSI scanout (GPU to panel):              1080p × 120 =      fixed 8.33 ms per frame
                                                   ───────────────────────────
TOTAL PER-FRAME (GPU time):              ~1.25 + 0.4 + 6.0 + 0.64
                                        = 8.3 μs compute
                                        vs 8.33 ms frame budget
                                        → GPU latency < 0.1% of frame time
                                        → 0.1% of frame = 8.3 μs, fits 60× over
```

> **Verification:** The GPU can render a full 1080p frame with 4× MSAA at 120 Hz with 99.9% of the frame budget remaining for driver overhead, V-Sync jitter compensation, and display blanking intervals. Thermal runaway is the only real-world limiter (see §10).

### 13.2 Power Efficiency Measurement (Mobile-SoC Typical)

```
Thermal Design Power (TDP) Budget
════════════════════════════════════
Total phone battery target: 5000 mAh @ 3.85 V = 19.25 Wh
7-hour SOT (screen-on time) = 19.25 Wh / 7 h ≈ 2.75 W average
GPU target: ≤ 35% of total SOT power (gaming-optimized phone)
  Max average GPU power during gaming: 2.75 W × 0.35 = 0.96 W

Pixelion-128 sustained @ 120 fps 1080p: ~430 mW (see §10.1 DVFS)
  → Remaining 530 mW = compositor + display + DRAM + modem + cooling pump
  → GPU is the second-largest consumer after display, but within budget

Peak (10 ms burst gaming): 510 mW @ 1800 MHz → 510 mW × 0.01 s = 5.1 mJ/frame
  vs sustained 430 mW × 8.33 ms = 3.58 mJ/frame
  42% higher energy per peak frame = acceptable for short gaming bursts
```
